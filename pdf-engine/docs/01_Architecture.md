# PDF Engine v2 Architecture

**Status:** Draft v2.0

---

# Vision

PDF Engine — это универсальный Executive Report Engine.

Его задача — преобразовывать структурированный AI Report JSON в профессиональные executive-отчеты независимо от источника данных.

Движок не зависит от MeetMind AI и должен поддерживать любые AI-продукты, генерирующие структурированные отчеты.

Основные принципы:

- один входной JSON;
- множество возможных представлений;
- единая архитектура рендеринга;
- адаптивная компоновка;
- отсутствие ручной настройки под конкретный отчет.

---

# Rendering Pipeline

Документ строится последовательно.

```
AI Report JSON
        ↓
Section Registry
        ↓
Container Builder
        ↓
Layout Engine
        ↓
Overflow Engine
        ↓
Renderers
        ↓
HTML
        ↓
PDF
```

### AI Report JSON

Единая структура отчета.

Является единственным источником данных.

---

### Section Registry

Определяет:

- какие секции существуют;
- какие данные относятся к каждой секции;
- является ли секция optional;
- порядок отображения.

---

### Container Builder

Группирует связанные секции.

Контейнер отвечает только за компоновку.

Секции не знают друг о друге.

---

### Layout Engine

Определяет:

- ширину секций;
- количество колонок;
- плотность;
- режим отображения.

---

### Overflow Engine

Следит за тем, чтобы документ помещался в доступное пространство.

При необходимости изменяет layout, density и размеры карточек.

---

### Renderers

Каждый Renderer отвечает только за отображение своей секции.

Renderer ничего не знает о соседних секциях.

---

# Design Principles

## Footer всегда существует

Footer является единственной обязательной секцией.

Пользователь не может отключить Footer.

---

## Все остальные секции optional

Любая содержательная секция может быть отключена пользователем.

Если данных нет — секция не отображается.

---

## Нет пустых секций

Движок никогда не резервирует место под отсутствующий контент.

Если секция пустая, она полностью исключается из документа.

---

## Statistics — это metadata

Meeting Statistics относится к документу, а не к содержимому встречи.

Поэтому она всегда отображается сразу после Header.

---

## Executive Summary всегда первая

Первой содержательной секцией всегда является Executive Summary.

После нее располагаются остальные блоки.

---

## Пользователь выбирает секции

Пользователь управляет набором отображаемых секций.

Порядок секций определяется движком.

---

## Layout зависит от данных

Размеры секций никогда не фиксируются заранее.

Layout рассчитывается автоматически исходя из объема информации.

---

## Container отвечает за layout

Container принимает решения:

- количество колонок;
- распределение ширины;
- скрытие пустых секций;
- перераспределение пространства.

Section отвечает только за собственный контент.

---

# Canonical Document Structure

```
Header

Meeting Statistics

Executive Summary

Key Metrics

Knowledge

    Insights
    Decisions
    Risks

Execution

    Tasks
    Owners

Architecture

Footer
```

Пользователь может отключать любые секции кроме Footer.

При отсутствии данных секция автоматически исключается из документа.

---

# Containers

## Knowledge Container

Содержит:

- Insights
- Decisions
- Risks

Особенности:

- горизонтальное размещение;
- автоматическое количество колонок;
- отсутствие пустых колонок;
- автоматическое перераспределение ширины.

Пример:

```
Insights | Decisions | Risks
```

Если Risks отсутствует:

```
Insights | Decisions
```

Если существует только Decisions:

```
Decisions
```

Если Insights значительно длиннее остальных секций, контейнер автоматически увеличивает его ширину.

---

## Execution Container

Содержит:

- Tasks
- Owners

Контейнер самостоятельно определяет оптимальную компоновку.

---

## Architecture Container

Содержит архитектурные блоки.

Количество карточек определяется входными данными.

---

---

Архитектурные карточки.

# Layout Rules

Движок обязан соблюдать следующие правила.

- не оставлять пустое пространство под отсутствующие секции;
- автоматически перераспределять ширину между соседними секциями;
- поддерживать различные режимы плотности отображения;
- стремиться размещать документ на одной странице;
- переносить информацию на вторую страницу только при физической невозможности разместить ее на первой.

---

# Extension Strategy

Добавление новой секции не должно требовать изменения ядра движка.

Новая секция должна регистрироваться через Section Registry и автоматически участвовать в процессе компоновки.

Примеры будущих секций:

- Timeline
- Participants
- Transcript
- Recommendations
- SWOT Analysis
- Financial Summary
- Action Plan
- AI Commentary

---

# Long-Term Goal

PDF Engine должен стать универсальным Executive Report Engine, способным строить:

- PDF Report;
- Web Report;
- Printable Report;
- Executive Canvas;
- Presentation View.

Все представления должны использовать единую архитектуру и единый входной AI Report JSON.
